# Migrating to v0.6.0

v0.6.0 reshapes how pipelines bind to deployment shape. The mental model that drove the v0.5 multi-target pattern — one Go entrypoint with N internal target shapes — is replaced by a clean split: **Go declares entrypoints (reusable work units); YAML enumerates pipelines (each one a single deployment shape)**. The two concepts had been awkwardly fused in `pipelines[].targets:` for a while; this release straightens them out.

If you have any repo with a `targets:` block in `.sparkwing/sparkwing.yaml`, this is a real migration. The framework rejects the legacy shape at parse time with a pointer to this guide.

If your repo had no targets and no per-job `OnTarget` calls, you can skip most of this guide: bump the SDK pin, re-run, ship.

## The new mental model

| Concept | Lives in | Job |
|---|---|---|
| **Entrypoint** | Go | A reusable work unit. Declares its arg types + schema. Knows nothing about runners, profiles, or deployment specifics. |
| **Pipeline** | YAML (`pipelines[]:`) | A named deployment binding. Picks an entrypoint, supplies defaults / guards / locked flags / dispatch metadata / triggers. |

One entrypoint can back many pipelines. Two pipelines can share the same entrypoint and differ only in their YAML policy. The CLI operator types `sparkwing run <pipeline-name>`; the framework resolves the pipeline → entrypoint binding from YAML at dispatch.

## What disappears

| v0.5 | v0.6 |
|---|---|
| `pipelines[].targets:` block | Split into N pipelines, each with its own `dispatch:` |
| `--target <name>` flag | Removed (pipeline name *is* the deployment selector) |
| `sparkwing.Target(ctx)` | Removed (use a regular pipeline arg if you want the value) |
| `sparkwing.Register[T](name, factory)` | Deprecated wrapper for `RegisterEntrypoint` + implicit binding |
| Profile `default-args:` | Removed (defaults live in pipeline YAML now) |
| `OnTarget("X")` on Job/WorkStep | API kept but inert; split into multiple pipelines instead |
| `Push.target:` / `Webhook.target:` trigger fields | Removed (one pipeline = one shape, including which trigger fires it) |
| `validateTargetSelection` orchestrator gate | Removed |
| `args.target:` (the unreleased v0.6 work-in-progress shape) | Removed before tag |

## What's new

- `dispatch:` block at pipeline level: runners, source, secrets, protected, approvals, backend. Same field set as the old `targets[name]:` value, lifted to the pipeline.
- `guards:` block: `require:` and `reject:` token lists evaluated at dispatch. Vocabulary: `profile-local`, `profile-controller`, `profile-name:<name>`, `arg:<flag>=<value>`.
- `defaults:` map: per-arg fallback values. Resolution priority: schema `Default()` < YAML `defaults:` < schema `Computed()` < operator CLI flag.
- `locked:` list: arg flag names the operator cannot override. CLI hard-errors with the locking pipeline's name.
- `sparkwing.RegisterEntrypoint[T](entrypointName, factory)`: registers a Go work unit. `BindPipelinesFromYAML(cfg)` walks pipelines.yaml and routes each pipeline name to its entrypoint.

## Recipe: split a `targets:` block into N pipelines

The mechanical part is the YAML rewrite. Before:

```yaml
pipelines:
  - name: deploy
    entrypoint: Deploy
    targets:
      dev:  {}
      prod:
        runners: [prod-pool]
        source: prod-secrets
        protected: true
```

After:

```yaml
pipelines:
  - name: deploy-dev
    entrypoint: Deploy
    description: "Deploy to dev (laptop default)"

  - name: deploy-prod
    entrypoint: Deploy
    description: "Deploy to production"
    dispatch:
      runners: [prod-pool]
      source: prod-secrets
      protected: true
    guards:
      reject: [profile-local]
    locked: [protected]
```

The Go side keeps one entrypoint (`Deploy`). The operator now types `sparkwing run deploy-prod` or `sparkwing run deploy-dev`; each pipeline gets its own `--help` page, its own flag set, its own policy.

If your Go code branched on `sparkwing.Target(ctx)` (`if target == "prod" { ... }`), pull the branch out: either move it into a YAML default that flips a regular arg, or — better — into different jobs that the two pipelines compose separately, or different entrypoints entirely.

## Recipe: declare typed args on a job

The v0.5 → v0.6 redesign retains the `WithArgs[T]` typed-args layer that was already in flight. Quick refresher:

```go
type DeployArgs struct {
    Image     string `desc:"OCI image ref"`
    Replicas  int    `desc:"replica count"`
    Protected bool   `desc:"require approval before applying"`
}

type Deploy struct {
    sparkwing.Base
    sparkwing.WithArgs[DeployArgs]
}

func (Deploy) Schema() (*sparkwing.Schema, error) {
    s := sparkwing.NewSchema[DeployArgs]()
    s.Field("Image").Required()
    s.Field("Replicas").Required().Range(1, 100)
    return s.Build()
}

func (j *Deploy) Work(w *sparkwing.Work) (*sparkwing.WorkStep, error) {
    return sparkwing.Step(w, "rollout", func(ctx context.Context) error {
        a := j.Args(ctx)
        // a.Image, a.Replicas, a.Protected
        return doRollout(a)
    }), nil
}

func main() {
    sparkwing.RegisterEntrypoint[DeployArgs]("Deploy", func() *Deploy { return &Deploy{} })
    sparkwing.Run()
}
```

Pipelines binding this entrypoint supply per-deployment defaults via YAML `defaults:`. Operators override with `--image=...` etc. The schema's `Computed(func(a Args) T)` can derive one arg from another (e.g. `Image` from `Profile`), keeping deployment-conditional logic in Go.

## Guard vocabulary

Tokens are flat strings; lists compose via AND inside `require:`, OR inside `reject:`. Pick a token from this set:

| Token | Matches when |
|---|---|
| `profile-local` | Active profile has no controller URL (laptop dispatch). |
| `profile-controller` | Active profile has a controller URL. |
| `profile-name:<name>` | Active profile's name equals `<name>`. |
| `arg:<flag>=<value>` | Resolved arg `<flag>` equals `<value>`. |

If you need a token that isn't in this set, add it to `pkg/pipelines/guards.go` rather than nesting predicates in YAML. The vocabulary is meant to stay flat.

Example: "this pipeline can only run against the prod profile, and only when the operator passes `--confirm true`":

```yaml
guards:
  require: [profile-name:prod, "arg:confirm=true"]
```

## Defaults priority

Resolution per arg, low to high:

1. Schema `Default(literal)` — the SDK author's safety net.
2. Pipeline YAML `defaults:` — the deployment binding's policy.
3. Schema `Computed(fn)` — derivation rule, function of already-resolved args. Runs above YAML defaults so a deployer who explicitly wants a derivation can still get it (the Go author asserts this knowingly).
4. Operator CLI `--flag` — explicit operator override always wins.

`Computed` sitting above YAML defaults is the one priority-ordering subtlety worth knowing: the YAML default supplies a *value*; `Computed` supplies a *rule*, and the SDK author who wrote the rule presumably had a reason.

## Locked flags

```yaml
locked: [protected, environment]
```

Operator runs `sparkwing run deploy-prod --protected=false` and gets:

```
--protected is locked by pipeline "deploy-prod"; remove the flag or use a different pipeline
```

Hard error, no silent ignore. The point is to make policy unmistakable.

## Job-level runner labels keep working

The job-side `Requires() []string{"macos"}` / `Prefers(...)` / `WhenRunner(...)` interfaces are unchanged. Pipelines that need a specific runner shape can layer pipeline-level `dispatch.runners:` on top; the orchestrator takes the union.

## `OnTarget` is inert

The `Job.OnTarget(...)` and `WorkStep.OnTarget(...)` chainable methods still compile so dogfood repos don't break at the build step. But there's no longer an "active target" the filter compares against, so any step declaring `OnTarget("X")` gets skipped at run time with "no target selected." Migrate by either:

- Splitting the pipeline into two that compose different job sets, OR
- Removing the `OnTarget` call entirely if the differentiation was cosmetic.

`OnTarget` is targeted for full removal in v0.7.

## Step-by-step migration recipe

For each repo with `targets:` blocks:

1. **Bump the SDK pin** (`sparkwing version update --sdk`). Run any tests. You'll get parse errors against the legacy YAML shape — that's the migration prompt.
2. **Split each `targets:` block into N pipelines.** Use the recipe above. Move per-target `runners` / `source` / `protected` / `backend` into the new pipeline's `dispatch:` block.
3. **Move `Push.target:` / `Webhook.target:` defaults** by putting the trigger directly on the pipeline you want it to fire (`deploy-prod.on.push:` instead of `deploy.on.push.target: prod`).
4. **Replace `sparkwing.Target(ctx)` calls.** Either drop them (the YAML / pipeline name is now the deployment selector) or read the value from a `WithArgs[T]` field if the step body actually needs it.
5. **Audit `OnTarget` calls.** Each one is now inert. Decide per-call whether to migrate to a pipeline split or just remove the call.
6. **(Optional) Adopt `RegisterEntrypoint` for new code.** Existing `Register` calls keep working through v0.6.x; new code can start using the cleaner API.
7. **Optionally add `guards:` / `locked:` / `defaults:`** to existing pipelines now that the surface exists.
8. **Run `sparkwing run pre-push`** before pushing the migration commit.

## Worked example: `release` → `release-all` + `release-one`

Before (one entrypoint, multi-target):

```yaml
- name: release
  entrypoint: Release
  targets:
    all:    { runners: [release-pool] }
    single: { runners: [release-pool] }
```

After (one entrypoint, two pipelines):

```yaml
- name: release-all
  entrypoint: Release
  defaults: { all: "true" }
  dispatch: { runners: [release-pool] }

- name: release-one
  entrypoint: Release
  dispatch: { runners: [release-pool] }
```

`sparkwing run release-all` and `sparkwing run release-one --component foo` are now distinct CLI invocations with distinct `--help` pages.

## Common gotchas

- **YAML parse error pointing here.** That's the framework refusing the legacy shape. Follow the message to the right replacement.
- **A pipeline with no YAML entry no longer dispatches.** v0.5 let you run any registered entrypoint by name. v0.6 requires every dispatchable pipeline to have a YAML entry (the registration alone isn't enough). The Register sugar wrapper keeps single-name legacy pipelines working — call sites that *don't* have a YAML entry will need one added.
- **`OnTarget` jobs silently skip.** If your run completes with "0 jobs ran," check whether the jobs that should have fired are declared with `OnTarget(...)`. The filter still runs; the active target is now always empty.
- **Profile `default-args:` parse error.** Remove the block from your profiles YAML; move the values into the pipeline YAML `defaults:` map.

## What didn't change

- `Plan` / `Job` / `Work` / `Step` / `Spawn` primitives.
- Job-level `Requires()` / `Prefers()` / `WhenRunner()` runner labels.
- The orchestrator's job DAG semantics, `Needs`, `Inline`, `RunAndAwait`, run-summary surface.
- Profile *resolution* (chain order, builtin laptop fallback). Profiles still answer "where dispatch goes"; they just don't carry per-arg defaults anymore.
- Triggers — including `push`, `webhook`, `schedule`, `pre_commit`, `pre_push`. Only the per-trigger `target:` defaulting field is gone.
- Secrets surface, `Config[T]` / `Secrets[T]` typed accessors.
