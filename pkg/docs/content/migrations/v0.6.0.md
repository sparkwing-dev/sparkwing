# Migrating to v0.6.0

v0.5.0 hard-cut the project YAML and profile model to a flatter shape with `--profile` as the only "where" flag and `--target` as the pipeline-internal deployment selector. v0.6.0 builds on that by generalizing `--target` into a typed-args system: jobs declare their own args via Go structs + an optional `Schema()` method; pipelines declare the schema-bearing `target` arg via a YAML `args:` block; profiles default any arg via `default-args:`. The CLI surface a pipeline exposes is now the union of every contained job's declared args, computed at registration time.

If you only had pipelines with no targets and no per-job typed args, your repo needs almost nothing: a one-line YAML rename for any `targets:` blocks and a CHANGELOG note. If you used `--target` everywhere or read `sparkwing.Target(ctx)` from step bodies, expect ~10 minutes per pipeline to migrate plus a re-test.

## `targets:` becomes `args.target:`

The pipeline-level `targets:` map is nested one level deeper under `args.target:`. Same value shape; the rewrite is mechanical:

1. At the pipeline's indent level, add an `args:` key.
2. Move the existing `targets:` block under `args:`, indented one level deeper, and rename it to `target:`.
3. Re-indent every line that was inside the original `targets:` block by two spaces.

That's it — the per-target value shape (`runners`, `source`, `secrets`, `protected`, `values`, `backend`) doesn't change.

**Before:**

```yaml
pipelines:
  - name: release
    entrypoint: Release
    targets:
      dev:  {}
      prod: { runners: [my-pool], source: prod-secrets, protected: true }
```

**After:**

```yaml
pipelines:
  - name: release
    entrypoint: Release
    args:
      target:
        dev:  {}
        prod: { runners: [my-pool], source: prod-secrets, protected: true }
```

**Why:** `args:` is the new home for every schema-bearing dispatch arg, not just `target`. Nesting `target` under `args.target` reserves the namespace for future bind names (which the SDK will accept as they ship). For v0.6.0, `target` is the only bind name the SDK recognizes; the loader rejects pipelines that declare an unknown one.

**Soft cut, not hard:** the loader still parses the legacy `targets:` block when `args:` is absent, so a v0.5-shaped repo keeps working unchanged. Declaring BOTH on the same pipeline errors at load time; that's the one combination the loader refuses, so an in-flight rename can't accidentally double-declare.

**Removal target:** v0.7.0. The legacy `targets:` field will be removed entirely then. Plan on landing the rename during the v0.6.x window.

## Optional: declare typed Args on your jobs

Jobs can opt into a typed `Args` struct via the new `WithArgs[T]` embedded helper. Doing so is what unlocks the rest of v0.6: declarative validators, profile defaulting, computed values, group constraints, the resolution chain that gates the run on required-when predicates. Jobs that don't embed `WithArgs[T]` keep working as plain function or `Workable`-implementing struct jobs.

**Minimal example:**

```go
type DeployArgs struct {
    Replicas int    `desc:"target replica count"`
    Image    string `desc:"OCI image ref"`
    DryRun   bool   `flag:"dry-run" desc:"skip the actual rollout"`
}

type DeployJob struct {
    sparkwing.Base
    sparkwing.WithArgs[DeployArgs]
}

func (j *DeployJob) Work(w *sparkwing.Work) (*sparkwing.WorkStep, error) {
    return sparkwing.Step(w, "rollout", func(ctx context.Context) error {
        a := j.Args(ctx)
        // a.Replicas, a.Image, a.DryRun -- typed reads inside the step body.
        _ = a
        return nil
    }), nil
}
```

`sparkwing run release --help` now lists `--replicas`, `--image`, `--dry-run` automatically (kebab-cased from the field name unless the `flag:"..."` tag overrides). Two struct tags carry their weight on every arg: `flag:"override-name"` (only when you don't want the default kebab-case mapping) and `desc:"help text"` (always, so `--help` and `pipeline describe` carry the explanation).

**Adding constraints** (Required, Default, Computed, OneOf, group rules) is a separate, optional `Schema()` method on the job. The schema is a tight Go expression — no template strings inside tags:

```go
func (DeployJob) Schema() (*sparkwing.Schema, error) {
    s := sparkwing.NewSchema[DeployArgs]()
    s.Field("Replicas").Required().Range(1, 100)
    s.Field("Image").RequiredWhen(sparkwing.Local)
    s.Field("DryRun").Default(false)
    return s.Build()
}
```

Jobs that have no constraints can skip `Schema()` entirely — the framework synthesizes a zero-constraint schema from the struct, so every field becomes a plain optional flag. You only write `Schema()` when you have something to say.

The full vocabulary (briefly): per-field `Required`, `RequiredWhen(predicate)`, `Default`, `Computed(fn)`, `DependsOn`, `Bind`, `OneOf`, `Min`/`Max`/`Range`/`Positive`, `Custom(fn)`; cross-field `Group(...).ExactlyOne()`/`.AtLeastOne()`/`.AtMostOne()`/`.AllOrNone()` with optional `.When(predicate)` and `.Desc(msg)` decorators; predicates `ArgEq`/`ArgNeq`/`ArgIn`/`ArgSet`/`ArgUnset`, combinators `And`/`Or`/`Not`, context predicates `Local`/`Remote`/`Profile("name")`/`Always`. See the doc comments on each for the exact contracts.

## Profile `default-args:` to default any arg

Profiles get a `default-args:` block to default any declared arg when that profile is active. Profiles **default** args; they never **declare** them — declaration always lives with the job or the pipeline-level `args:` block.

```yaml
# ~/.config/sparkwing/profiles.yaml
default: prod
profiles:
  prod:
    controller: https://api-sparkwing.rangz.dev
    token: swu_...
    default-args:
      target: prod                # --profile prod -> target=prod implied
      version: "0.6.1"
  ci:
    detect: { env_var: CI, equals: "true" }
    default-args:
      target: staging
      version: ${CI_BUILD_VERSION}   # ${ENV} interpolation supported
```

Resolution order per arg, first match wins: explicit `--flag` → profile `default-args:` → schema `Computed` (function of already-resolved args) → schema `Default` (literal). Then `RequiredWhen` predicates fire, validators check the resolved value, group constraints fire last.

Interpolation: only `${VAR}` placeholders are recognized in `default-args:` values; richer shell syntax (POSIX `${VAR:-default}`, `$(cmd)`, bare `$VAR`) is rejected at parse time with a clear error naming the offending key.

## `sparkwing.Target(ctx)` is deprecated but still works

The single-purpose `sparkwing.Target(ctx)` accessor that v0.5 added to read the resolved `--target` value is still present in v0.6, marked deprecated. It now wraps the more general `sparkwing.Arg[string](ctx, "target")`. The v0.5-shaped callers keep compiling; new code should reach for `Arg[T]`:

```go
// v0.5 (still works in v0.6, deprecated)
target := sparkwing.Target(ctx)

// v0.6 (preferred)
target, err := sparkwing.Arg[string](ctx, "target")
if err != nil { return err }
```

The deprecation removes the function in v0.7.0. Find call sites with `git grep 'sparkwing\.Target('` and migrate at your leisure during the v0.6.x window.

## `--target` is no longer a first-class flag

In v0.5, `--target` was a sparkwing-defined CLI flag the framework parsed before handing off to the pipeline. In v0.6, `--target` is just *one* of the args a pipeline can declare via `args.target:` in YAML or via a struct field with `Bind("target")` on a job. The CLI exposes it only when the active pipeline declares it.

For pipelines that have an `args.target:` block: `--target X` keeps working exactly as before. The framework wires the bound runners / source / secrets / etc. from the YAML when it resolves the target arg's value.

For pipelines that declare no targets: `--target` no longer appears in `--help`, and passing it errors with "unknown flag" instead of "no target named X." That's the intended consequence — pipelines that don't have target shape shouldn't pretend to.

The "multi-target requires `--target`" gate from v0.5 generalizes: when any arg has 2+ allowed values, no default, and the resolver can't pick one, the error message names the arg and points at the resolution-chain step that fired ("arg `target` declares multiple values [dev,prod] and no value provided; pass `--target` or set profile `default-args.target`").

## Step-by-step migration recipe

For each repo with a `.sparkwing/sparkwing.yaml`:

1. **Apply the `targets:` → `args.target:` rewrite** described in the section above. One-time edit per pipeline that had a `targets:` block. If both shapes accidentally end up co-existing, `sparkwing pipeline list` fails loudly with the offending pipeline's name.

2. **(Optional) Adopt `WithArgs[T]` for jobs that want typed args.** Touch only the jobs that benefit; leaving the rest as plain function jobs is fine. Start with the job that has the most CLI flags today (the biggest declarative win) and grow from there.

3. **(Optional) Move `--target` defaults into profile `default-args:`.** If your operators repeatedly type `--target prod` against the prod profile, lift it into the profile so the flag becomes optional in that context.

4. **Find and replace `sparkwing.Target(ctx)` call sites** with `sparkwing.Arg[string](ctx, "target")` — at your pace; the old call keeps working through v0.6.x.

5. **Run `sparkwing run pre-push`** in the repo before pushing the migration commit. The pre-push gate catches mod-tidy drift from the SDK bump and any lint regressions.

## Common gotchas

- **Flag name collisions across jobs in one pipeline.** Two jobs each declaring `Replicas int` will collide on `--replicas` and the plan registration panics with a message naming both. Disambiguate via a `flag:"override-name"` tag on one or both — the framework doesn't auto-prefix in v0.6.

- **`RequiredWhen` referencing an unresolved arg.** Predicates reference args by their CLI flag name (kebab-cased). If you write `RequiredWhen(ArgEq("Target", "prod"))` (Go field name, not flag name) the predicate never matches because `Target` isn't a key in the resolved map. Use lowercase `"target"`.

- **Profile `default-args:` for an arg the pipeline doesn't declare.** Currently produces no error, only a runtime debug log. A typo (`target` vs `targt`) silently has no effect. We may tighten this in v0.6.x with an opt-in strict mode; until then, run `sparkwing profile` to inspect resolved args and catch typos.

- **`bind:` other than `"target"`.** Reserved for future schema-bearing arg kinds (`runner`, `profile`, others). Declaring `bind:"runner"` today errors at schema build time. Stick to `bind:"target"` for v0.6.

## What didn't change

- The Plan / Job / Work / Step / Spawn primitives. Pipelines and jobs you wrote against v0.5 keep their existing dispatch model; the args layer wraps the inputs side.
- The orchestrator's job DAG semantics. `Needs()`, `Inline()`, `RunAndAwait`, the run-summary surface — all unchanged.
- Profile resolution chain order (flag → project hint → detect → default → builtin laptop). `default-args:` extends the per-profile content but doesn't reorder how profiles themselves get picked.
- `pkg/pipelines.Target`'s field set. The legacy `Target` struct that `targets:` was unmarshaling into is reused unchanged for the new `args.target:` values; both shapes parse to the same type.
