# v0.8 release plan: make hidden runtime state explicit (proposal)

**Status:** scope agreed. `Verify` and the probe helper are at
proposal grade; observable gates needs a design pass. Two items await
sign-off from field review. Not yet implemented.

## Theme

One idea ties the release together: **surface runtime state that is
currently implicit.** Today two things happen silently:

- A step's command exits 0 while the thing it deployed is unhealthy --
  there is no signal between "command ran" and "result is good."
- A step waiting on a busy concurrency slot is indistinguishable from a
  hang -- you cannot see that it is queued, behind whom, or for how long.

v0.8 makes both explicit. Everything additive; no breaking changes.

## Workstreams

### 1. `Verify` postcondition gate -- core SDK

**Status:** designed. See [verify-postcondition-gate](verify-postcondition-gate.md).

A node-attached lifecycle modifier checked after the action succeeds; a
failed check fails the node with stage provenance, so "the mutation
failed" and "the mutation completed but its postcondition failed" become
distinguishable.

Surface (SDK):

- `Verify(fn func(ctx) error)` on `JobNode`.
- `Failure{Stage, Err}`, `FailureStage` (`StageAction` / `StageVerify`).
- `OnFailure` extended to accept `func(ctx, Failure) error`.

Decided, in brief: no `OnVerifyFailure` sugar (one explicit
stage-aware hook); core never learns HTTP/auth/health meanings; cache
means cache (a cached node skips action + verify); the
verify-as-node + `RunIf` engine is rejected (recorded in the proposal).

Optional observability refinement (on-theme; small): a recovery callback
may return `sw.RecoveryResult{Action, Reason, Err}`, which the
orchestrator records into the run ledger and emits as an event -- so logs
and the dashboard show *which* branch fired and *why*
("recovery chose rollback; reason verify_unhealthy"). `Action` and
`Reason` are opaque author-defined strings; core does not interpret them,
preserving the rule that it never learns health semantics. Available in
degraded form today via plain logging in the recovery closure, so it is a
refinement, not a blocker.

### 2. `sparks-core/probe` -- spark library (not the SDK)

**Status:** designed. Independent timeline; lands in the existing
`sparks-core` optional module, next to `deploy` / `kube` / `gitops`.

The HTTP health probe that produces a `func(ctx) error` for `Verify`.
Generic `net/http` with nothing sparkwing-specific, so it stays out of
the base SDK.

Surface (library):

- `probe.HTTP(url)` builder: `Method`, `Header`, `HeaderFunc`
  (per-attempt credentials), `Body`, `ExpectStatus`, `ExpectJSON`,
  `Retry`, `Interval`, `Timeout`, `.Check`.
- Error classification: `probe.Indeterminate(err) bool` so recovery can
  roll back on a definitively-unhealthy response but escalate when the
  check merely could not run (expired token, unreachable endpoint).

### 3. Observable concurrency gates -- runtime + dashboard

**Status:** scoped, design to finalize. Folds into the event model the
[host-admission](host-admission.md) proposal already sketches.

When a node waits on a full concurrency slot (`OnLimit: Queue`), it
blocks with no external signal. The ask: surface the current holder and
an ahead-of count so a wait reads as "queued for `<ns>`: N ahead, held
by `<run>`" instead of an indistinguishable hang.

Sketch (to be finalized against the concurrency / `AcquireSlot` path):

- Stream events mirroring host-admission's shape, keyed by namespace:
  `slot_queued {namespace, position, holder, waited_ms}`,
  `slot_granted {namespace, waited_ms}`.
- A synchronous lookup keyed by namespace so a wrapper / the dashboard
  can render queue state without per-process polling.

This workstream is independent of 1 and 2 and can proceed in parallel;
it needs its own short design pass before it is proposal grade.

## Sequencing

- **1 then 2's payoff:** the probe helper is only useful once `Verify`
  exists, but the two can be built in parallel; 2 just cannot ship
  *valuable* before 1.
- **3 is independent** of 1 and 2.
- **The deploy cluster** (`sparks-core/deploy.Verified` returning a
  `JobGroup`) is the first consumer that proves 1 + 2 together. It needs
  no SDK change beyond `Verify` and is library work, so it rides 2's
  timeline.

## Out of scope / deferred (with reasons)

- **`OnVerifyFailure` sugar.** A shortcut spelled "on verify failure do
  X" nudges toward "verify failed -> rollback," which is unsafe (the
  indeterminate case). Add later if the explicit branch proves painful;
  easier to add API than remove it.
- **verify-as-node + `RunIf`/`Failed`/`Succeeded`/`Output` engine.**
  Rejected -- larger core surface, duplicates `OnFailure`/`SkipIf`/`Ref`,
  fights the lifecycle-modifier model, reintroduces dishonest node
  status. Unifying the dependency model is its own future proposal.
- **Cluster primitive + `plan.Scope("name")` auto-namespacing.** Clusters
  ship now as plain `sparks-core` functions returning a `JobGroup`; the
  only friction is manually prefixing internal node IDs per instance.
  One cluster is a function; several with copy-pasted prefixing justify
  the `Scope` affordance. Defer until that evidence exists.
- **Declarative `sw.Recover{OnActionFailure, OnVerifyUnhealthy,
  OnVerifyUnknown}` struct.** The three-way form requires core to
  distinguish unhealthy from indeterminate -- which is the probe's
  classification, deliberately kept out of the SDK. A boundary-safe
  two-way form would still branch internally on `probe.Indeterminate`,
  so it buys little over the closure. Same bucket as `OnVerifyFailure`:
  possible later sugar, not v0.8.
- **Canary / traffic-split.** Needs a substrate a single-instance
  topology lacks.
- **Richer `Verify` + `.Cache()` interaction.** "Cache means cache" is
  the v0.8 rule; a narrower rule (allow verify-on-hit, forbid only
  auto-recovery-on-hit) waits until cache-hit-verify semantics are
  designed.

## Already shipped (this cycle, ahead of v0.8)

- **docs:** `deployment-modes.md` blesses running the hosted controller
  single-instance on SQLite, so solo operators don't think Postgres is
  required.

## Open items needing sign-off

- **Two reversals of the field review**, both flagged in the `Verify`
  proposal: dropping `OnVerifyFailure` (the review wanted it primary),
  and simplifying the cache rule from a build-time refusal to "cache
  means cache." Neither is likely contentious -- the first follows from
  the review's own auth-token example -- but they are the reviewer's to
  confirm.
- **Observable-gates final design** (workstream 3): the concrete event /
  lookup surface, after reading the concurrency path.

## What "ship it" looks like

- Workstream 1: the SDK changes and tests enumerated in the `Verify`
  proposal; SDK-reference and pipelines-guide docs; CHANGELOG `**sdk:**`
  entry under v0.8.
- Workstream 2: the `sparks-core/probe` package and tests; released on
  `sparks-core`'s own version line.
- Workstream 3: events + lookup + dashboard wiring; tests for the
  contention/queue path; CHANGELOG `**controller:**` / `**cli:**` entry.
- A v0.8 entry in CHANGELOG.md and, if any surface turns out breaking
  (none expected), a `docs/migrations/` section.
