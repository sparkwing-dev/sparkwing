# v0.6 pipeline redesign: entrypoints vs pipelines

## Status

Implemented and shipped to main. Replaces the earlier v0.6 work
(typed args + `args.target:` YAML block + profile `default-args:`)
with a cleaner factoring before tagging v0.6.0.

## Post-implementation revisions

After the initial implementation landed, two decisions changed:

1. **`locked:` removed.** The "args the operator cannot override"
   concept conflated two unrelated mechanisms. Args are by
   definition operator-controllable; values the operator should
   not control belong in `PipelineConfig` (the `values:` block).
   The migration guide steers callers to that mechanism instead.
2. **Defaults priority reordered.** YAML `defaults:` now beats
   schema `Computed()`, not the other way around. The invariant:
   *explicit* values (deployer YAML, operator CLI) beat
   *fallback rules* (SDK-author `Default`/`Computed`). The
   deployer who wrote a YAML value did so intentionally; silently
   overriding it with a Computed rule was surprising. Final
   priority, low to high: `Default` -> `Computed` -> `defaults:`
   -> CLI.

## Motivation

The v0.6 work so far solved the right problem with the wrong shape. We generalized `target` into an arg but kept the multi-target single-pipeline pattern, which forced:

- Profile `default-args:` to silently shift pipeline behavior on profile switch (defaults split across versioned YAML and unversioned per-operator config).
- The `args.target:` YAML block to fuse two concerns: declaring an arg with enumerated values, AND binding each value to dispatch metadata (runners, source, secrets, protected, backend) the SDK can't reach.
- `OnTarget("prod")` to live alongside the generalized predicate vocab as parallel-purpose mechanisms.

The redesign is structural, not additive: pipelines stop being "one Go entrypoint plus N target shapes" and become "one YAML deployment binding pointing at a Go entrypoint." Multi-target pipelines decompose into multi-pipeline YAML; the Go side becomes radically smaller and infra-name-free.

## The mental model

**Entrypoint (Go).** The work unit. Knows what work exists, declares arg types and validators via `Schema()`, declares dependencies between jobs, declares intrinsic runner constraints (e.g. `Requires() []string{"macos"}`). Knows nothing about where it runs, what profiles to allow, what defaults are sensible in any given environment. Reusable across pipelines and across repos.

**Pipeline (YAML).** Names the entrypoint, supplies per-deployment policy: defaults, guards, locks, dispatch routing, triggers. Each pipeline is one "shape of run" -- same entrypoint can back many pipelines, each with its own name and policy. A pipeline must have a YAML entry to be invocable.

## Concrete shape

### Go side

```go
type DeployArgs struct {
    Image     string `desc:"OCI image ref"`
    Replicas  int    `desc:"replica count"`
    Protected bool   `desc:"require approval before applying"`
}

type Deploy struct {
    sparkwing.Base
    sparkwing.WithArgs[DeployArgs]
}

func (Deploy) Schema() (*sparkwing.Schema, error) {
    s := sparkwing.NewSchema[DeployArgs]()
    s.Field("Image").Required()
    s.Field("Replicas").Required().Range(1, 100)
    return s.Build()
}

// Intrinsic runner constraint: this work needs a macos-labeled runner.
// Pipeline YAML can layer additional Requires on top.
func (Deploy) Requires() []string { return []string{"macos"} }

func (j *Deploy) Work(w *sparkwing.Work) (*sparkwing.WorkStep, error) {
    return sparkwing.Step(w, "rollout", func(ctx context.Context) error {
        a := j.Args(ctx)
        return doRollout(a)
    }), nil
}

func main() {
    // Note the API change: RegisterEntrypoint, not Register.
    sparkwing.RegisterEntrypoint[DeployArgs]("Deploy", func() *Deploy { return &Deploy{} })
    sparkwing.Run()
}
```

The Go code has zero references to `prod`, `dev`, `prod-pool`, `prod-secrets`, profile names, or any deployment specifics. Forking the entrypoint across orgs requires no edits.

### YAML side

```yaml
# .sparkwing/sparkwing.yaml
pipelines:
  - name: deploy-prod
    entrypoint: Deploy
    description: "Deploy to production"

    guards:
      require: [profile-controller]
      reject:  [profile-local]

    defaults:
      image:    "registry.prod.com/myapp:latest"
      replicas: "10"

    locked: [protected]

    dispatch:
      runners:   [prod-pool]
      source:    prod-secrets
      protected: true

    triggers:
      - on: tag
        pattern: "v*"

  - name: deploy-dev
    entrypoint: Deploy
    description: "Deploy to dev"
    defaults:
      image:    "nginx:latest"
      replicas: "1"
    # No guards, no dispatch, no locks. Runs wherever invoked.

  - name: release-all
    entrypoint: Release
    defaults: { all: "true" }

  - name: release-one
    entrypoint: Release
    # operator picks the component via flags
```

## Vocabulary

### Guards (pipeline-level dispatch gates)

Flat predicate tokens. The framework rejects dispatch (with a clear error naming the failed guard) when `reject:` matches or `require:` doesn't match.

Known tokens, v0.6:

- `profile-local` -- active profile has no controller URL.
- `profile-controller` -- active profile has a controller URL.
- `profile-name:<name>` -- active profile's name equals `<name>`.
- `arg:<flag>=<value>` -- resolved arg `<flag>` equals `<value>`.

Tokens compose via lists: `require: [a, b]` means both must match (logical AND). `reject:` is rejection-on-any-match (logical OR). For more elaborate logic, add a token rather than introducing nested predicate trees in YAML.

### Locked

`locked: [flag1, flag2, ...]` -- lists arg flag names the operator cannot override via `--flag` or env. The CLI errors hard when the operator passes a locked flag: `--protected is locked by pipeline deploy-prod; remove the flag or use a different pipeline.` Silent override would be worse than refusal.

### Defaults

Per-arg fallback values, keyed by flag name. Type coercion mirrors the schema's: strings parse via the field's type. Resolution priority, low to high:

1. Schema `Default()` (SDK-author's safety net)
2. Pipeline YAML `defaults:` (deployment-binding default)
3. Schema `Computed(fn)` (function of already-resolved args)
4. Explicit `--flag` from CLI / env (operator override)

(`Computed` sits above YAML defaults because it's expressing a derivation rule, not a default value. If the deployer wants to force a literal that beats Computed, they pass `--flag` explicitly.)

### Dispatch

The block of metadata the framework needs at scheduling time. Today's `Target` field set, lifted up to the pipeline level:

- `runners: [name, ...]` -- runner pool / label names.
- `source: <name>` -- secret source backend.
- `secrets: <name>` -- alternate secret resolver.
- `protected: true` -- refuses dispatch from non-default branches.
- `backend: { ... }` -- per-pipeline cache/log/state override.

Absent `dispatch:` block means "laptop default": in-process runner, no source binding, no protection gate.

### Triggers

Unchanged from current pipelines.yaml shape. One pipeline = one set of triggers; if you want one entrypoint triggered two ways, declare two pipelines.

## API change: RegisterEntrypoint

Today: `sparkwing.Register[T](name, factory)` couples name + entrypoint + pipeline at Go registration time.

v0.6: `sparkwing.RegisterEntrypoint[T](entrypointName, factory)` registers the work unit by its Go-side name (matching the YAML `entrypoint:` field). Pipelines come from YAML. The runtime resolves `sparkwing run deploy-prod` by reading the YAML, finding `entrypoint: Deploy`, fetching the registered `Deploy` factory.

`Register[T]` stays as a deprecation-marked sugar wrapper that registers an entrypoint AND inserts an implicit pipeline entry for laptop-default dispatch. Removal in v0.7.

## What disappears

- `args.target:` YAML block (and the v0.5 `targets:` block it nests).
- `pipelines[].targets` field in `pkg/pipelines.Pipeline`.
- `pipelines[].Args` (the schema-bearing per-arg YAML map we added in v0.6).
- Profile `default-args:` block + `internal/profile/default_args.go`.
- `Target` arg as a framework convention; `--target` flag; `sparkwing.Target(ctx)` accessor (already deprecated).
- `OnTarget(...)` on Job/WorkStep -- collapses into pipeline-level gating via separate pipelines.
- `validateTargetSelection` orchestrator gate.
- `ProfileResolutionContext.Defaults` field.
- The `target` mirror at the orchestrator boundary.

Roughly 700 lines of code across SDK + orchestrator + CLI + tests.

## What stays

- Profiles (still "where dispatch goes": controller URL, token, state/logs/cache routing).
- Predicate vocab (`ArgEq`/`Local`/`Remote`/`Profile(name)` etc.) -- now uniformly used for guards, defaults, and step-level `When` rather than scattered across target validation, OnTarget, and predicate-driven schemas.
- Schema vocab (`Required`/`Default`/`Computed`/`OneOf`/`Min`/`Max`/etc.).
- `WithArgs[T]` job-level typed args.
- Job-level runner constraints (`RequiresProvider`/`PrefersProvider`/`WhenRunnerProvider`). Pipeline `dispatch.runners:` layers on top -- pipeline-declared runners are added to the job's intrinsic constraints, not replaced.
- Triggers, jobs.yaml, sparks.yaml.
- Job-level gating via `When(predicate)` (already exists for step bodies).

## Migration

Affected: `moonborn-ws`, `moonborn-web`, `sparkwing-product` (the three repos with `targets:` blocks today).

For each pipeline with N targets, split into N pipelines manually:

**Before:**

```yaml
pipelines:
  - name: deploy
    entrypoint: Deploy
    targets:
      dev:  {}
      prod: { runners: [prod-pool], source: prod-secrets, protected: true }
```

**After:**

```yaml
pipelines:
  - name: deploy-dev
    entrypoint: Deploy

  - name: deploy-prod
    entrypoint: Deploy
    dispatch:
      runners: [prod-pool]
      source: prod-secrets
      protected: true
    guards:
      reject: [profile-local]
```

Go code: drop any `sparkwing.Target(ctx)` reads. If the pipeline branched on target value (`if target == "prod" { ... }`), that branch typically wants to become a flag the operator controls, OR a different entrypoint, OR (for genuine policy differences like "prod uses prod-bucket") a YAML `defaults:` entry that propagates the value into a flag the Go code already reads.

No automated migration tool. The three-repo migration is a couple hours by hand and produces cleaner YAML than a mechanical converter would.

## Implementation order

1. **YAML parser** -- add `entrypoint`, `guards`, `defaults`, `locked`, `dispatch` fields to `pkg/pipelines.Pipeline`. Drop the `Targets` / `Args` (v0.6) fields. Back-compat: keep `Targets` parsing through v0.6.x with a deprecation warning logged at load time so the three dogfood repos keep working until they migrate.
2. **Guard evaluator** -- internal/profile + sparkwing predicate context, evaluates flat token strings against the resolved profile + args.
3. **Lock enforcer** -- CLI-side flag parser rejects locked flags with the named error.
4. **Defaults resolver** -- orchestrator merges YAML `defaults:` into `ResolveInputs.FlagValues` (priority: schema default < YAML default < computed < CLI). Drops `ResolveInputs.ProfileDefaults`.
5. **`RegisterEntrypoint` API** -- split out of `Register`. Keep `Register` as deprecation-marked wrapper.
6. **Orchestrator integration** -- drops the `Target` field flow, drops `validateTargetSelection`, drops `--target` flag handling, drops the args-mirror at the invoke boundary.
7. **Doc rewrite** -- `docs/migrations/v0.6.0.md` becomes the migration story for this redesign instead of the args.target one. `pkg/docs/content/` resync.
8. **CHANGELOG entries** -- replace the unreleased entries we already wrote.

Estimated total: 2–3 focused days.

## Open questions

None at draft time -- the four open questions from the chat thread (predicate vocab shape, lock semantics, defaults priority, bare-entrypoint invocation) are answered above.

## Non-goals

- Job-level guards ("this job cannot compose into a pipeline with profile-local"). Punted -- additive feature when an author actually needs to enforce composability constraints.
- Repo-level cross-pipeline defaults. The user explicitly rejected this in favor of per-pipeline `defaults:`.
- A YAML predicate DSL richer than flat tokens. If a token isn't expressive enough, add a token rather than nesting.
